Boris — runbook (Pi-1)
======================

Boris runs as four systemd --user services on pi-1, started at boot
via loginctl linger:

  boris-bridge.service     Go bridge (whatsmeow → SQLite + REST on :8080)
  boris-bot.service        Python admin_bot.py (polls SQLite, calls Claude)
  boris-watchdog.timer     fires boris-watchdog.service every 5 min
  boris-canary.timer       fires boris-canary.service daily at 10:15

Unit files live at ~/.config/systemd/user/ on the Pi. The Go bridge
binary lives at ~/projects/whatsapp-mcp/whatsapp-bridge/whatsapp-bridge
(separate clone of verygoodplugins/whatsapp-mcp); a symlink at
~/projects/tennis-pairings/whatsapp-mcp points there so admin_bot.py's
hard-coded path resolves.

Auto-recovery story
-------------------

bridge + bot have Restart=always with a 30s / 15s back-off and a
StartLimit cap of 20 attempts per 10 min. They self-heal from a
crash within ~30 s.

The bridge unit's ExecStartPre waits up to 60 s for DNS to resolve
web.whatsapp.com before launching the binary — covers the "service
started before the network was ready" race after a power-failure
reboot. Independent of that wait, Restart=always (not on-failure)
also catches the bridge's "tried 3 times to connect, gave up,
exited 0" path, which used to leave us dead after a boot race.

The watchdog (scripts/boris_watchdog.sh) is belt-and-braces for the
cases systemd can't see — bridge process up but HTTP wedged, or
unit parked in "failed" state after exhausting StartLimit. Every
5 min it:

  - probes http://127.0.0.1:8080/ (any HTTP response = alive)
  - restarts boris-bridge if down for 2 consecutive checks
  - restarts boris-bot if systemctl says not active
  - posts a heads-up into the admin group when the bridge
    recovers after an outage >= 5 min (so the team knows there
    was a gap; ADMIN_GROUP_JID in .env)
  - greps the bridge journal for "Client outdated (405)" — the
    error WhatsApp returns when the whatsmeow library is too old.
    If found, touches a sentinel at /run/user/1000/boris-bridge-
    locked-out and logs at ERROR severity. We can't notify via
    WhatsApp in that state (the bridge IS what's locked out), so
    journalctl -p err is how it surfaces.

Tail the watchdog with:

    journalctl --user-unit=boris-watchdog -f

State across runs lives at /run/user/1000/boris-watchdog.state and
clears on Pi reboot, which is fine — a fresh boot is implicitly a
"recovered from down" event handled by the unit hardening above.

If the watchdog says "bridge LOCKED OUT", the recovery is to update
whatsmeow + rebuild the bridge:

    ssh pi-1
    export PATH=$PATH:/usr/local/go/bin
    cd ~/projects/whatsapp-mcp/whatsapp-bridge
    go get -u go.mau.fi/whatsmeow@latest
    go mod tidy && go build -o whatsapp-bridge .
    systemctl --user restart boris-bridge

Independent of the watchdog, boris-canary.timer fires once a day
(10:15 local) and runs scripts/boris_daily_canary.py. The canary
sends "Boris daily test msg" to the Boris admin group via the
bridge's /api/send, then watches messages.db for the echo. If the
round trip fails the service exits non-zero — `systemctl --user
--failed` will show boris-canary.service. The bot recognises the
exact marker string and silently drops it so the chat stays clean
(see CANARY_MARKER in admin_bot.py). Tail with:

    journalctl --user-unit=boris-canary -f

Manual canary run:

    ~/projects/tennis-pairings/.venv/bin/python \
        ~/projects/tennis-pairings/scripts/boris_daily_canary.py

Connect (over Tailscale)
------------------------

    ssh pi-1


Status / start / stop / restart
-------------------------------

    systemctl --user status  boris-bridge boris-bot
    systemctl --user start   boris-bridge boris-bot
    systemctl --user stop    boris-bot boris-bridge      # stop bot first
    systemctl --user restart boris-bot                   # most common


Live logs
---------

    journalctl --user-unit=boris-bot    -f
    journalctl --user-unit=boris-bridge -f

Use --since "10 min ago" or -n 200 for past output.


Sanity check
------------

Send `boris hi` from your phone in either:
  - Thursday tennis Admin
  - Boris the tennis bot

Boris should reply within ~5 seconds, prefixed with
"From Boris the tennis bot: ".


WhatsApp QR re-link (when the linked-device session expires, ~14 days
of phone inactivity)
---------------------------------------------------------------------

The bridge prints a QR code in its journal on startup if re-link is
needed. To scan it:

    systemctl --user stop boris-bridge
    cd ~/projects/whatsapp-mcp/whatsapp-bridge
    ./whatsapp-bridge        # QR appears in this terminal — scan from
                             # WhatsApp -> Settings -> Linked Devices
                             # -> Link a Device, then Ctrl+C
    systemctl --user start boris-bridge


Notes
-----

- session_state.json (current evening's attendees + courts) is NOT
  cleared on restart. If you want a fresh start say `boris clear
  tonight` before the first command.
- Per-command Anthropic API usage is logged to the "Usage log" tab of
  the Google Sheet. Authoritative billing is at
  https://console.anthropic.com/.
- Auto-kickoff: every Thursday at 09:35 BST/local, the bot fires
  thursday_kickoff once. Tracked in .kickoff_state.json so it never
  double-fires.
- Code updates: `git pull` on the Pi, then
  `systemctl --user restart boris-bot` (and bridge too if Go side
  changed).
